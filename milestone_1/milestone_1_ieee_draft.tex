\section{Milestone 1: Technical Understanding and Research Framing}

\subsection{Topic}

\begin{itemize}
    \item The topic of this seminar paper is the impact of loop unrolling on controller delay in high-level synthesis.
    \item The main reference paper studies how different loop unrolling factors influence the delay of the controller generated during high-level synthesis.
    \item The paper focuses on the problem that loop unrolling can improve performance, but can also increase the complexity and delay of the finite-state machine controller.
\end{itemize}

\subsection{Problem Statement}

\begin{itemize}
    \item Loop unrolling is a common optimization technique.
    \item It duplicates the body of a loop so that multiple loop iterations are executed together.
    \item In software compilation, loop unrolling is usually used to reduce loop overhead and expose more instruction-level parallelism.
    \item In high-level synthesis, loop unrolling has additional hardware effects.
    \item The unrolled loop body becomes part of the generated hardware structure.
    \item This can increase the number of operations, control states, multiplexers, and resource-sharing decisions.
    \item Because of this, the controller delay can increase.
    \item If controller delay is not considered, the synthesized design may fail to meet the required clock period.
    \item Therefore, the problem is not only how to reduce latency, but how to choose an unroll factor that improves performance while still satisfying timing constraints.
\end{itemize}

\subsection{Motivation}

\begin{itemize}
    \item High-level synthesis allows designers to create hardware from high-level descriptions such as C or C++.
    \item This makes hardware development easier, but it also means that compiler-style optimizations need to be understood from a hardware perspective.
    \item A transformation that looks beneficial at the algorithmic level may create timing or resource problems after synthesis.
    \item Loop unrolling is a good example of this issue.
    \item A larger unroll factor can reduce the number of loop iterations.
    \item However, it can also increase the size and delay of the controller.
    \item This makes it important to study the tradeoff between latency reduction and controller-delay increase.
\end{itemize}

\subsection{Core Technical Idea}

\begin{itemize}
    \item The main idea is to predict how the controller delay changes when the loop unroll factor changes.
    \item Instead of synthesizing every possible unroll factor, the paper proposes an estimation-based method.
    \item The method uses high-level information such as:
    \begin{itemize}
        \item number of operations,
        \item number of control steps,
        \item number of FSM states,
        \item number of state bits,
        \item number of functional-unit control bits,
        \item number of variables.
    \end{itemize}
    \item These values are used to estimate FSM/controller delay.
    \item The estimated delay is then used to guide the selection of a suitable unroll factor.
    \item The goal is to find an unroll factor that improves latency while remaining inside a given delay budget.
\end{itemize}

\subsection{Important Technical Concepts}

\begin{itemize}
    \item High-Level Synthesis:
    \begin{itemize}
        \item Converts a behavioral description into RTL hardware.
        \item Generates a datapath and a controller.
    \end{itemize}

    \item Datapath:
    \begin{itemize}
        \item Contains arithmetic units, registers, multiplexers, and memories.
    \end{itemize}

    \item Controller:
    \begin{itemize}
        \item Usually implemented as a finite-state machine.
        \item Controls when operations happen and which resources are selected.
    \end{itemize}

    \item Loop Unrolling:
    \begin{itemize}
        \item Duplicates loop body operations.
        \item Reduces loop-control overhead.
        \item Exposes more parallelism.
        \item Can increase hardware complexity.
    \end{itemize}

    \item Controller Delay:
    \begin{itemize}
        \item Delay caused by FSM logic and control decisions.
        \item Can become part of the critical path.
        \item Can limit the achievable clock period.
    \end{itemize}
\end{itemize}

\subsection{Formalism and Model}

\begin{itemize}
    \item The paper uses an estimation model for FSM delay.
    \item A simplified form of the model is:
\end{itemize}

\begin{equation}
Delay = A \cdot \log(\#operations) \cdot (\#state\ bits)
\end{equation}

\begin{itemize}
    \item The constant $A$ depends on hardware-specific factors such as functional-unit control bits and number of variables.
    \item The number of operations increases when the loop body is duplicated.
    \item The number of state bits depends on the number of FSM states or control steps.
    \item The unroll factor affects both operation count and controller complexity.
\end{itemize}

\subsection{Assumptions}

\begin{itemize}
    \item The method assumes that the loop structure is suitable for unrolling.
    \item The loop bounds and unroll factors must be known or estimable.
    \item The delay model depends on the synthesis flow and target technology.
    \item The method assumes that controller delay can be estimated from high-level information.
    \item The paper focuses on controller delay, not the complete timing behavior of all possible hardware targets.
\end{itemize}

\subsection{Scope Limitations}

\begin{itemize}
    \item The original experiments were performed using a specific behavioral synthesis framework and ASIC library.
    \item The exact numerical delay values may not directly transfer to modern FPGA tools.
    \item Modern HLS tools may apply additional optimizations that interact with loop unrolling.
    \item Controller delay may not always be directly visible in student-level HLS reports.
    \item In a student implementation, resource usage, latency, clock estimate, and timing slack may need to be used as indirect indicators.
\end{itemize}

\subsection{Resource, Runtime, and Scalability Aspects}

\begin{itemize}
    \item Exhaustively synthesizing every possible unroll factor can be very time-consuming.
    \item The number of possible design choices grows when multiple loops are considered.
    \item Large unroll factors can increase:
    \begin{itemize}
        \item operation count,
        \item FSM complexity,
        \item multiplexer size,
        \item functional-unit sharing complexity,
        \item synthesis time,
        \item hardware area.
    \end{itemize}
    \item The paper proposes estimation and pruning to avoid unnecessary synthesis runs.
    \item This makes the approach more scalable than exhaustive exploration.
\end{itemize}

\subsection{Open Questions}

\begin{itemize}
    \item How accurately can the original delay model be reproduced using modern FPGA-based HLS tools?
    \item Can controller delay be separated clearly from datapath and routing delay in student-level synthesis reports?
    \item How does loop unrolling interact with loop pipelining in modern HLS tools?
    \item What is the best low-budget way to demonstrate the tradeoff experimentally?
    \item Would a Python simulation be sufficient, or should a small HLS experiment also be included?
\end{itemize}
